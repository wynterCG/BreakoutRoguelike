The Big Picture

Think of the game as a team of specialists. Each script has one clear job, and they communicate by sending signals (like passing notes).

MonsterData (.tres files on disk) | loaded by | main.gd ----creates----> block.gd ----uses----> health_component.gd | | | | ball hits it emits "died" | | | | hit() called block removes itself | | +---- listens for "destroyed" signal ---------------+ | +---- counts remaining blocks | if 0 left -> "YOU WIN!" | +---- kill zone detects ball falling past paddle lose a life, or "GAME OVER" if 0 lives paddle.gd <---- player input (A/D or Left/Right arrows) | ball.gd ---- bounces off paddle, walls, blocks | kill zone ---- ball fell past paddle = lose a life

Godot Glossary

Key Godot and programming concepts used throughout the codebase, explained in plain language.

extends

Means "this script is a type of" something. extends CharacterBody2D means "this script is a type of physics body that I can move with code." It inherits all the built-in behavior of that type.

class_name

Gives your script a global name so other scripts can refer to it. class_name Ball lets any script say var b: Ball instead of loading the file manually.

signal

An announcement system. A script can emit (shout) a signal, and other scripts can connect (listen) for it. Like a doorbell — the button doesn't know who's inside, it just rings.

@export

Makes a variable show up in the Godot editor's Inspector panel. You can tweak values visually without touching code. Like a settings dial on the outside of a machine.

@onready

Grabs a reference to a child node, but waits until the scene is fully loaded first. @onready var _paddle = $Paddle is like saying "once everything's set up, remember where the paddle is."

@tool

Makes a script run inside the Godot editor, not just during gameplay. Used for editor plugins and tools that help you build the game.

_ready()

A built-in function that Godot calls once when the node first enters the scene. Used for initialization — setting up shapes, colors, connecting signals.

_physics_process(delta)

Called every physics frame (60 times per second). Use this for movement and collision. delta is the time since the last frame (~0.016 sec), used to keep movement smooth regardless of framerate.

CharacterBody2D

A physics body you move with code. It doesn't float or fall on its own — you tell it exactly where to go. Used for the paddle and ball where we want precise, arcade-style control.

StaticBody2D

A physics body that doesn't move but other things can collide with it. Used for blocks/enemies — they sit still and the ball bounces off them.

Resource

A data container that can be saved to disk as a .tres file. Think of it like a config file or a "character sheet" — it holds values but isn't a game object itself.

PackedScene

A .tscn file loaded into memory, ready to be stamped out as copies. preload() loads it at startup. .instantiate() creates a new copy. Like a cookie cutter — load once, stamp many.

Vector2

Two numbers bundled together: X (horizontal) and Y (vertical). Used for positions, directions, sizes. Vector2(100, 200) means "100 pixels right, 200 pixels down."

queue_free()

Tells Godot "delete this node at the end of the current frame." Safer than instant deletion because other code might still be referencing it this frame.

.connect()

Wires a signal to a function. button.pressed.connect(my_func) means "when this button is pressed, call my_func." It's how scripts talk to each other without knowing about each other.

.emit()

Fires a signal. Anyone connected to it will have their function called. destroyed.emit() means "announce that I was destroyed."

Why a Resource?

A Resource is Godot's way of storing data in a file. Instead of hardcoding "Brute has 3 HP and is red" in the code, we put that in a .tres file. This means you can add, edit, and delete monster types without touching any code — just edit the file in the Godot inspector or use the Monster Editor dock.

HealthComponent Component

res://scripts/components/health_component.gd

A reusable HP tracker. Instead of writing health logic inside every enemy, we write it once here and attach it as a child node to anything that needs HP.

Signals (announcements it can make)

SignalWhen it firesWho listens
health_changedEvery time HP goes up or downBlock — updates the HP bar visual
diedWhen HP reaches 0Block — emits "destroyed" and removes itself

Functions

FunctionWhat it does
initialize(hp)Sets starting HP. Called once when the block is created.
take_damage(amount)Subtracts HP (default: 1). Uses maxi() to prevent going below 0. Emits health_changed, and if HP hits 0, also emits died.
get_current_hp()Returns the current HP number. Other scripts call this to know how much HP is left.
is_alive()Returns true if HP > 0.

Full code

extends Node
class_name HealthComponent

signal health_changed(current_hp: int, max_hp: int)
signal died

@export var max_hp: int = 1
var _current_hp: int = 0

func _ready() -> void:
    _current_hp = max_hp

func initialize(hp: int) -> void:
    max_hp = hp
    _current_hp = hp

func take_damage(amount: int = 1) -> void:
    _current_hp = maxi(_current_hp - amount, 0)
    health_changed.emit(_current_hp, max_hp)
    if _current_hp <= 0:
        died.emit()

func get_current_hp() -> int:
    return _current_hp

func is_alive() -> bool:
    return _current_hp > 0

Why a Component?

The "component pattern" means building features as separate, attachable pieces. Instead of every enemy having its own HP code (copy-pasted and hard to maintain), they all share one HealthComponent. When we later add enemies that heal or have shields, we only change this one file.

Paddle Gameplay

res://scripts/paddle.gd

The player-controlled arc paddle. Moves left/right with keyboard input. Its curved shape is the core aiming mechanic — where the ball hits determines which direction it bounces.

Constants

NameValueWhat it controls
MOVE_SPEED600Pixels per second of horizontal movement
ARC_WIDTH200Total width of the paddle in pixels
ARC_HEIGHT30How deep the curve is (taller = more extreme angles)
ARC_SEGMENTS16How many straight lines make up the curve (more = smoother)
SCREEN_MARGIN10Pixels of padding from screen edges

How the arc shape works

_generate_arc_points() creates the curved paddle by placing 17 dots along a parabola (U-shape), then connecting them into a polygon:

Imagine drawing this shape, point by point:

   \                          /      ← edges curve up (steep angles)
    \                        /
     \______________________/        ← center is flat (vertical bounce)

Then close the bottom with a flat line to make a solid shape.

How aiming works

get_bounce_normal(hit_position) is the most important function for gameplay. When the ball hits the paddle, it calculates which direction to bounce it based on where on the paddle it landed:

Hit the left edge   → ball bounces up-left   (steep angle)
Hit the center       → ball bounces straight up (vertical)
Hit the right edge  → ball bounces up-right  (steep angle)

It uses the derivative (slope) of the parabola at the hit point to find the surface angle, then calculates a perpendicular direction (the "normal"). This is the math that makes the ball reflect realistically off the curve.

Movement

Every physics frame (60fps), the paddle:

  1. Reads input: Input.get_axis("move_left", "move_right") returns -1 (left), 0 (nothing), or +1 (right)
  2. Sets velocity: direction × 600 pixels/sec, horizontal only
  3. Calls move_and_slide() — Godot's built-in "move and handle collisions"
  4. Clamps position so the paddle can't go off-screen

Ball Gameplay

res://scripts/ball.gd

The bouncing projectile. Starts sitting on the paddle, launches when you press Space, then bounces around hitting blocks. The ball is what deals damage to enemies.

Two states

The ball is always in one of two modes, controlled by the _is_launched boolean:

StateBehaviorTransitions to
On paddle (false)Follows paddle position, 40px above it. Waits for Space press.Launched (press Space)
Launched (true)Flies in a direction at 400 px/sec, bounces off everything.On paddle (falls into kill zone with lives remaining)

The physics frame (every tick)

Pseudocode of _physics_process():

if not launched:
    stick to paddle
    if Space pressed: launch upward with slight random angle
    return

# Ball is in flight:
move in current direction at current speed
if we hit something:
    if it's the paddle: bounce using the paddle's arc-based angle
    else: bounce using simple reflection (angle in = angle out)

    # Safety: prevent horizontal-only bouncing (boring infinite loop)
    if vertical speed too low: force minimum vertical movement

    # Deal damage
    if the thing we hit has a hit() method: call it

Key details

move_and_collide() vs move_and_slide()

The paddle uses move_and_slide() which handles collisions automatically. The ball uses move_and_collide() which stops at the first collision and gives you collision info so you can calculate the bounce yourself. We need manual control because the ball has custom bounce logic.

MIN_VERTICAL_RATIO safety net

Set to 0.3. If the ball's vertical speed component drops below 30% of its total speed, it gets forced back up. Without this, the ball could get stuck bouncing horizontally between walls forever — boring and unwinnable.

How hit() calling works

if collider.has_method("hit") — this checks if whatever we bounced off has a function called hit(). Walls don't have one (so nothing happens). Blocks do (so they take damage). This is a "duck typing" approach: "if it looks like it can be hit, hit it."

Block (Enemy Brick) Gameplay

res://scripts/block.gd

The enemy. A static brick that takes hits from the ball. Has an HP bar that shrinks as it takes damage, and a number showing remaining HP. Reads its stats from a MonsterData resource.

Visual structure (3 layers)

[Dark background] — always full-size, dark gray (#262626). This is the "empty" part of the bar.
[Colored fill bar] — shrinks left-to-right as HP decreases. Color comes from MonsterData.
[HP number] — white text centered, shows current HP with dark shadow for readability.

Setup flow (_ready)

1. Read stats from MonsterData (or use fallback defaults)
2. Initialize HealthComponent with HP
3. Create collision rectangle matching block size
4. Set up dark background (full size of block)
5. Set up colored fill bar (inside background, with 2px padding)
6. Set up HP label (white text, centered, with shadow)
7. Connect to HealthComponent signals

Damage flow

Ball hits block
      ↓
hit() called
      ↓
_health.take_damage()
      ↓
HealthComponent emits health_changed
      ↓
_update_visual() runs:
  - Calculates hp_ratio = current / total   (e.g., 2/3 = 0.666)
  - Shrinks fill bar width proportionally    (fill = inner_width * 0.666)
  - Updates HP number text                   ("3" → "2")
      ↓
if HP = 0: HealthComponent emits died
      ↓
_on_died(): emit destroyed signal, then queue_free() (delete self)

Fallback system

If no MonsterData is assigned (e.g., during testing), the block falls back to hardcoded colors based on HP tier: 1 HP = green, 2 HP = yellow, 3 HP = red. This keeps the game playable even without any .tres files.

Main (Game Manager) Manager

res://scripts/main.gd

Startup sequence (_ready)

2. _load_monster_types()     ← Read all .tres files from data/monsters/
3. _spawn_enemies()          ← Create 14×5 grid of bricks
4. _spawn_ball()             ← Create ball on the paddle
5. _update_ui()              ← Show "Lives: 3"
6. Connect kill zone signal  ← Detect ball falling past paddle

How monster loading works

_load_monster_types() opens the res://data/monsters/ folder and reads every .tres file into a list. Then it sorts by HP descending — toughest first. This way, row 0 (top) gets the 3 HP Brutes, row 1 gets 2 HP Warriors, and rows 2-4 get 1 HP Grunts.

How spawning works

for row in 0..4:
    for col in 0..13:
        create a new Block from the block.tscn template
        assign monster type based on row number
        position it on the grid using spacing math
        listen for its "destroyed" signal
        add it to the game
        increment block counter

Win/lose logic

EventWhat happens
Block destroyedCounter decreases. If 0 blocks left → "YOU WIN!"
Ball enters kill zoneLose a life. If lives > 0 → reset ball to paddle. If lives = 0 → "GAME OVER"
Game over + SpaceReload the entire scene (fresh restart)

Editor Plugin Editor Tool

res://addons/monster_editor/plugin.gd

The plugin entry point. It does one thing: when enabled, it loads the Monster Editor dock and adds it to the Godot editor. When disabled, it removes it.

Full code

@tool
extends EditorPlugin

const DOCK_SCENE_PATH: String = "res://addons/monster_editor/monster_editor_dock.tscn"
var _dock: Control = null

func _enter_tree() -> void:
    var scene: PackedScene = load(DOCK_SCENE_PATH) as PackedScene
    if scene:
        _dock = scene.instantiate()
        add_control_to_dock(DOCK_SLOT_RIGHT_BL, _dock)

func _exit_tree() -> void:
    if _dock:
        remove_control_from_docks(_dock)
        _dock.queue_free()
        _dock = null

Why load() instead of preload()?

preload() loads a file at compile time — if the file doesn't exist yet, the script fails to compile. load() loads at runtime, which is safer for plugins where the dock scene might not exist when the plugin script is first parsed by Godot.

Game Loop

What happens every single frame (60 times per second) while the game is running.

Every frame (~16ms):

1. INPUT PHASE
   Read keyboard: is Left/Right/A/D pressed? Is Space pressed?

2. PADDLE UPDATE
   Calculate velocity from input direction × 600
   Move paddle horizontally
   Clamp to screen edges

3. BALL UPDATE
   if ball is on paddle:
       Snap ball position to 40px above paddle center
       if Space pressed: launch with random slight angle
   else (ball is flying):
       Move ball: position += direction × speed × delta
       if collision detected:
           if hit paddle: calculate bounce angle from arc shape
           else: simple reflection
           if too horizontal: force vertical minimum
           if hit something with hit(): call it (deals damage)

4. BLOCK REACTIONS (triggered by ball hits)
   Block.hit() → HealthComponent.take_damage()
   → health_changed signal → update HP bar and number
   → if HP = 0: died signal → destroyed signal → queue_free()

5. GAME STATE CHECK
   if block destroyed: decrement counter
       if counter = 0: "YOU WIN!"
   if ball in kill zone: lose life
       if lives = 0: "GAME OVER"
       else: reset ball to paddle

6. RENDER
   Godot draws everything to screen automatically

Roadmap

Where the project stands and what's planned next.

Phase 1: Core Breakout Loop

Arc paddle, ball physics, walls, breakable blocks, lives system, win/lose. The basic playable game.

Phase 2: Monster Data System

MonsterData resources replace hardcoded blocks. HP bar visuals with fill + number. Editor dock plugin for managing monster types.

Phase 3: "Ouch Zone" Mechanic

Balls never leave play. Missed balls bounce off a back wall behind the paddle but deal damage to the paddle. Run ends at 0 Paddle HP.

Phase 4: Upgrades

Extra balls, piercing, fire trails, splitting, homing, knockback, wider paddle, extra HP, dash, shield, magnetic pull.

Phase 5: Roguelike Systems

Procedural room generation, branching map, run progression, currency.

Phase 6: Polish & Balancing

VFX, sound, game feel, difficulty tuning, UI polish.
